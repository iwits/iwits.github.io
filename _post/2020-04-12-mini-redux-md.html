# 为什么需要Redux?
  • react 是一个单向数据流的结构,整个react应用的数据智能单向想下分发或者自行内部消化,react 不具备数据向上回溯的功能.因此当我们需要祖父向孙子,兄弟组件,子向祖先组件之间大量通信的时候.我们需要使用一个状态管理库来辅助我们对项目进行管理.
  • 前端复杂性的根本原因就是大量无规律的交互和异步操作,redux 尝试让每一个state的变化都是可以预测的,将应用中所有的动作与状态统一管理.

什么是redux?

redux 的特点

  • redux是一个可以预测状态变化的状态管理库(著名的状态管理库还有mobx等).他具有三个显著的特性

    • 单一数据源(store)

    • 数据的state为只读

    • 必须要通过纯函数(dispatch(action){})来修改数据的state

redux的原理(基本流程)

redux的工作流程

  1. 首先View通过dispatch来触发事件action

  2. reducer会根据action.type 对事件进行处理,返回一个state

  3. store会保存reducer产生的新数据,并触发subscribe订阅的事件

  4. 最后,view根据getState()获取到最新的数据,进行展示

  5. 重复以上步骤.

    从这里我们可以看出来action(动作)\ store(存储)\reducer就是三个核心

手写实现一个简单的redux

createStore(reducer)

  • store 作为存储单一数据源的地方.暴露了四个重要的方法

    • getState() : 为了读取state提供的方法,返回结果是最新的state
    • dispatch(action): dispatch 接受一个action.将当前的State和对应的action告知reducer,要求reducer返回一个新的state.[为什么全新的state?因为必须用纯函数来实hexo现reducer,避免副作用,如果直接修改原来的state,那么整个函数就不能称之为纯函数]
      • 调用reducer(curState, action)
      • 按照顺序执行curlisteners;
      • 返回一个action
    • subscribe(newListener): 内部干了两件事情:1.一个函数添加订阅. 2.直接返回一个unSubscribe函数.[源码中整个部分写法,我感觉非常优雅,避免了再内部使用复杂的数据结构来维护listener的列表].
    • replaceReducer(nextReducer): 为了热替换\按需加载准备的一个函数,
      • curReducer = nextReducer
      • dispatch({type: ActionTypes.INIT})

  • store 内的数据结构是

    • curReducer = reducer
    • state
    • curListeners = [];
    • nextListeners = curListeners;
